# 作業進捗管理

## 基本原則：1ページ = 1Issue

各用語ページに対して1つのIssueを作成し、作業履歴・状態管理をそのIssue内で一元管理する。

---

## Issueタイトルルール

**良い例:**
- `api - テンプレート準拠チェック`
- `cpu - SEO改善`
- `dns - コンテンツ拡充`

**悪い例（避ける）:**
- `[SEO] APIページの改善` ← タイトルにラベル情報を入れない
- `【最優先】CPUページ修正` ← 優先度はラベルで管理
- `（緊急・SEO）DNSページ` ← カッコ付き文言は不要

---

## Issueラベル体系

| ラベル | 用途 |
|--------|------|
| `SEO` | 検索エンジン最適化（title/description/キーワード） |
| `description` | title・descriptionのメタ情報 |
| `コンテンツ` | 記事の執筆・品質管理 |
| `UI/UX` | デザイン・使いやすさ |
| `next` | Next.js実装タスク |
| `仕組み作り` | ワークフロー・ルール・テンプレート整備 |
| `bug` | 不具合 |

## ページの状態管理

各ページは「内容」「SEO」の2観点で管理する。状態はIssue内のチェックリストで記録。

### テンプレート準拠の状態
- **✅ 完璧**: 概要・デモ・FAQ・関連用語の順序が正しい + 人間が実際に見て「完璧」と判断
- **🔁 改善待ち**: テンプレート構造の改善が必要なページ

### SEO状態
- **✅ 適切**: title/description/キーワードが適切に設定されている
- **🔁 改善待ち**: SEOチェックで問題が見つかったページ

### Issueの構造（テンプレート）
```markdown
**作成日**: YYYY-MM-DD

## 📊 ページの現状
**最終更新**: YYYY-MM-DD

### テンプレート準拠
- [ ] ✅ 完璧（概要・デモ・FAQ・関連用語の順序完璧 + 人間が確認して完璧と判断）
- [ ] 🔁 改善待ち（テンプレート構造の改善が必要）

### SEO状態
- [ ] ✅ 適切（title/description/キーワードが適切）
- [ ] 🔁 改善待ち（SEOチェックで問題発見）

---

## 🔁 課題
[なぜこの作業が必要か、何が問題か]

---

## 💻 実施内容
- [ ] タスク1
- [ ] タスク2
- [ ] タスク3

---

## ✅ 完了条件
- [ ] 条件1
- [ ] 条件2
```

### 作業履歴の記録方法
**重要**: 作業履歴はIssue本文ではなく、**コメント**に記録する

**記録例:**
```
2026-02-06: コンテンツ拡充完了（2,500→3,600文字）
- FAQ 6個追加
- 実例 4個追加
- デモ説明を200文字拡充

✅ 完璧に更新
```

**メリット:**
- 時系列で自動的に並ぶ
- 議論の流れが残る
- GitHubから通知が来る
- 本文がすっきり

### 状態の更新ルール
1. **新規ページ作成時**: Issue作成 → テンプレート準拠チェック → Issue内に結果を記録
2. **人間が完璧と判断**: Issueコメントに「✨ 完璧に更新」と記録 → チェックボックス更新
3. **新タスク追加時**: ✨ を取るか確認 → 必要なら「🔁 改善待ち」に戻す → コメントに理由を記載
4. **SEOチェック時**: ✨ の有無を確認 → 改善が必要か判断 → Issueコメントに記録

## 作業の流れ

### フェーズ1: 新規ページ作成
1. `/new-page [用語名]` 実行
2. テンプレート準拠チェック
3. Issue作成（タイトル: `[用語名] - 初回作成`）
4. チェック結果をIssue内に記録

### フェーズ2: 人間による確認
1. 人間が内容を確認
2. 問題があれば Issue内に問題点を記載 → 「作業待ち」タグ
3. 問題なければ Issue内に「内容: 完璧 ✨」と記録

### フェーズ3: 継続的な改善
1. SEOデータをチェック（Search Console）
2. `/seo-check` で自動修正
3. 同一Issue内に修正内容を追記
4. 完璧 ✨ タグの有無を再確認

### フェーズ4: Issue更新のタイミング
- SEO改善時: 同一Issue内に追記
- コンテンツ拡充時: 同一Issue内に追記
- デザイン変更時: 同一Issue内に追記

**原則: 1ページの全作業履歴は1つのIssueに集約する**

---

## Issueクローズのタイミング

以下の条件を満たしたらIssueをクローズ:
- 作業が完了した（チェックリストがすべて完了）
- 現時点で追加作業の予定がない

**新たな問題が発生したら、新しいIssueを作成する**
- 例: 「api - テンプレート準拠チェック」が完了 → クローズ
- 後日SEO問題が発覚 → 新しく「api - SEO改善」Issueを作成

**メリット:**
- Issueが肥大化しない
- 作業履歴が時系列で追いやすい
- 完了したタスクとオープンなタスクが明確

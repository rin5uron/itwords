# 📅 更新日・最新情報確認の対応ポリシー

**ITは鮮度が命！** 古い情報は信頼を失うため、更新日と最新情報確認の対応ルールをまとめます。

---

## 🎯 基本ルール

### ✅ 必ず更新日を入れる
すべてのページに `dateModified` を設定し、コンテンツ更新時には必ず日付を更新する。

```tsx
<StructuredData
  type="Article"
  title="..."
  description="..."
  datePublished="2024-01-09"
  dateModified="2026-01-16"  // ← 更新日を必ず最新に
/>
```

---

## 📋 ページ種別ごとの対応

### 🔴 最優先：時間経過で陳腐化する情報

変更される可能性が高い情報を含むページは、**最新情報確認の注意書き**を必ず追加する。

#### 対象ページ
| ページ | 陳腐化リスク | 対応内容 |
|--------|------------|---------|
| **Vercel** | 🔴 高 | 料金プラン変更、機能追加 |
| **npm** | 🔴 高 | コマンド変更、バージョン管理 |
| **GitHub** | 🟡 中 | UI変更、機能追加 |
| **Framework** | 🟡 中 | 人気フレームワーク変動 |
| **Hosting Service** | 🟡 中 | 価格・機能変更 |
| **API** | 🟡 中 | REST API仕様変更 |

#### 注意書きテンプレート

**料金・プラン系**（Vercel、Hosting Serviceなど）：
```tsx
<p className="note">
  <strong>⚠️ 注意：</strong>
  [サービス名]の料金プランは変更されることがあります。
  最新の情報は<a href="[公式URL]" target="_blank" rel="noopener noreferrer">公式サイト</a>で確認してください。
  <br />
  <small>（最終更新：YYYY年MM月DD日）</small>
</p>
```

**技術仕様系**（npm、APIなど）：
```tsx
<p className="note">
  <strong>⚠️ 注意：</strong>
  [技術名]の仕様は変更されることがあります。
  最新の情報は<a href="[公式ドキュメントURL]" target="_blank" rel="noopener noreferrer">公式ドキュメント</a>で確認してください。
  <br />
  <small>（最終更新：YYYY年MM月DD日）</small>
</p>
```

---

### 🟡 中優先：定期的に見直すべき情報

変更頻度は低いが、時々確認が必要なページ。

#### 対象ページ
| ページ | 見直し頻度 | チェック内容 |
|--------|----------|------------|
| **JavaScript** | 6ヶ月に1回 | 新しい構文、ブラウザ対応 |
| **CSS** | 6ヶ月に1回 | 新しいプロパティ、ブラウザ対応 |
| **HTML** | 6ヶ月に1回 | 新しいタグ、廃止タグ |
| **Cookie** | 6ヶ月に1回 | ブラウザのCookie規制変更 |
| **LocalStorage** | 1年に1回 | ブラウザ仕様変更 |

#### 対応内容
- 半年〜1年ごとに内容を見直し
- `dateModified` を更新
- 大きな変更があれば注意書き追加

---

### 🟢 低優先：普遍的な概念

変更されにくい基礎概念。更新日の管理は緩めでOK。

#### 対象ページ
- **Stack** - データ構造は変わらない
- **Queue** - データ構造は変わらない
- **Validation** - 概念は普遍的
- **Poka-yoke** - 製造業の概念
- **Foolproof** - 概念は普遍的
- **AES** - 暗号化アルゴリズム（標準規格）
- **Zero-day** - セキュリティ概念
- **Exploit** - セキュリティ概念
- **White Hat Hacker** - セキュリティ概念
- **CVE** - 脆弱性管理システム

#### 対応内容
- コンテンツ追加時のみ `dateModified` 更新
- 注意書き不要

---

## 🔄 更新日を更新すべきタイミング

### ✅ 必ず更新する
- コンテンツを追加・修正したとき
- デモを追加・改善したとき
- FAQを追加・修正したとき
- 料金・仕様が変更されたとき

### ❌ 更新しなくていい
- タイポ修正のみ
- CSSスタイル調整のみ
- 内部リンク追加のみ
- メタデータ（title/description）のみの変更

## 💡 ポイント

1. **ユーザーの信頼が第一**
   - 古い情報をそのまま放置しない
   - 「最新情報は公式で確認してね」と誘導する

2. **更新日は正直に**
   - 小さな修正でも実質的な内容変更なら更新日を更新
   - タイポ修正だけなら更新しなくていい

3. **陳腐化リスクを見極める**
   - 料金・仕様 → 🔴 高リスク → 注意書き必須
   - 概念・基礎 → 🟢 低リスク → 更新日のみでOK

4. **定期チェックを習慣化**
   - 3ヶ月に1回はリスク高いページを確認
   - 1年に1回は全ページを見直し
